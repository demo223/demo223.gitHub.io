# 5.2、@Transactional的使用

@Transactional 注解，它其中有很多配置，称为事务属性，摘录如下

属性名称 含义 补充说明
propagation 传播行为 -
isolation 隔离级别 -
timeout 超时时间 当规定时间内，事务仍未完成时，会回滚并结束事务
readOnly 是否只读 某些数据库驱动会对只读事务做优化
rollbackFor 遇到什么异常回滚事务 -
noRollbackFor 遇到什么异常不回滚事务 -

  1. 一个事务如果是做查询操作，可以设置为只读，此时数据库可以针对查询操作来做优化，有利于提高性能。

    @Transactional(readOnly = true)
    1

  2. 一个数据库操作有可能因为网络或死锁等问题卡住很长时间，从而导致数据库连接等资源一直处于被占用的状态。所以我们可以设置一个超时属性，让一个事务执行太长时间后，主动回滚。事务结束后把资源释放出来。

    @Transactional(timeout = 60) //单位为秒
    1

  3. 在代码编写过程中，编译期程序需要抛出异常，运行之后发现事务仍然提交了。Spring 默认只有抛出运行时异常（即 RuntimeException 及子类）或 Error 及子类时，才会回滚事务，如果抛出的是其它编译时异常，仍然会提交事务。

    @Transactional(rollbackFor = Exception.class)
    1

最终查看数据库发现虽然出现了异常，但事务仍然提交了。

原因：

  • Spring 默认只有抛出运行时异常（即 RuntimeException 及子类）或 Error 及子类时，才会回滚事务
  • 如果抛出的是其它编译时异常，仍然会提交事务

解决办法：

方法1：配置 rollbackFor = Exception.class只要出现任何异常(Exception 或 Exception 的子类异常)就回滚。

方法2：手动try、catch异常，然后在catch代码块里面抛出异常

@Override
@Transactional
public void transfer(int fromId, int toId, double amount) {
	//开始进行转账
	try {
		accountMapper.updateAmount(fromId, -amount);
		new FileInputStream("aaa");	//	模拟编译时异常
		accountMapper.updateAmount(toId, amount);
	} catch (Exception e) {
		throw new RuntimeException(e);	//	手动抛出运行时异常，可以使事务回滚
	}
}

方法2：在方法上面配置@SneakyThrows注解

@SneakyThrows会自动帮你try、catch异常，但是最后会抛出异常。

@SneakyThrows会将当前方法抛出的异常，包装成RuntimeException，骗过编译器，使得调用点可以不用显示处理异常信息。

在加入该注解后，控制台最后抛出的异常信息为

	java.lang.reflect.UndeclaredThrowableException
1

UndeclaredThrowableException是RuntimeException的子类，因此最终事务不会提交，会事务回滚。

            this.accountMapper.updateAmount(toId, amount);
        } catch (Throwable var6) {
            throw var6;
        }
    }

# 5.3、事务传播

问题：如果在ServiceA的a方法中调用了ServiceB的b方法，以谁的事务为准？

# 事务传播行为

属性名称 含义 补充说明
REQUIRED 需要事务，有则加入，无则创建新事务
REQUIRES_NEW 需要新事务，无论有无，总是创建新事务
SUPPORTS 支持事务，有则加入，无则在独立的连接中运行 SQL 结合 Hibernate、JPA 时有用，配在查询方法上
NOT_SUPPORTED 不支持事务，不加入，在独立的连接中运行 SQL -
MANDATORY 必须有事务，否则抛异常 -
NEVER 必须没事务，否则抛异常 -
NESTED 嵌套事务 仅对 DataSourceTransactionManager 有效

两个受事务控制的业务方法之间进行调用时，事务是合二为一、还是另开新事务，这可以通过传播行为进行控制，对其中最常用的 required 和 requires_new 说明如下

ServiceA.foo() 调用 ServiceB.bar()，其传播行为组合测试如下 测试1

  • foo - @Transactional(propagation = Propagation.REQUIRED)
  • bar - @Transactional(propagation = Propagation.REQUIRED)
  • 结论 foo bar 同一事务

测试2

  • foo - @Transactional(propagation = Propagation.REQUIRED)
  • bar - @Transactional(propagation = Propagation.REQUIRES_NEW)
  • 结论 foo bar 不同事务

测试3

  • foo - @Transactional(propagation = Propagation.REQUIRED)
  • bar - @Transactional(propagation = Propagation.REQUIRES_NEW)
  • bar - 出现异常、foo 不捕获
  • 结论 bar 回滚，foo 也会回滚

测试4

  • foo - @Transactional(propagation = Propagation.REQUIRED)
  • bar - @Transactional(propagation = Propagation.REQUIRES_NEW)
  • bar - 出现异常、foo 捕获
  • 结论 bar 回滚，foo 因为捕获了异常，因此会提交

# 5.4、事务失效问题

情况1：编译期异常导致事务失效

  • 在代码编写过程中，编译期程序需要抛出异常，运行之后发现事务仍然提交了。Spring 默认只有抛出运行时异常（即 RuntimeException 及子类）或 Error 及子类时，才会回滚事务，如果抛出的是其它编译时异常，仍然会提交事务。

  • 解决办法：配置 rollbackFor = Exception.class

情况2：捕捉异常导致事务失效

  • 经常有同学喜欢自己捕捉异常，在业务层被事务管理的方法内我们的异常能不能捕获？不能，如果try-catch了会发现最终事务 还是提交了。事务提交、回滚，都是由代理对象调用事务通知方法来完成的，如果代理对象不知道出现了异常，也就没有机会去执行回滚，自己 try-catch异常，意味着代理对象认为没有发生异常，因此也会提交事务。
  • 解决办法：业务方法内不要捕获异常、或者将捕获的异常重新抛出。

@Service
public class ServiceC {

    private static final Logger log = LoggerFactory.getLogger(ServiceC.class);

    //	自己注入自己
    @Autowired
    private ServiceC serviceC;

    @Transactional(propagation = Propagation.REQUIRED)
    public void foo() {
        log.debug("foo");
        serviceC.bar();	//	这里就是走了代理
    }

  
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void bar() {
        log.debug("bar");
    }
}
