# SQL索引

# 一、MySQL索引创建原则

# 1.选择唯一性索引

唯一性索引的值是唯一的，可以更快速的通过该索引来确定某条记录。例如，学生表中学号是具有唯一性的字段。为该字段建立唯一性索引可以很快的确定某个学生的信息。如果使用姓名的话，可能存在同名现象，从而降低查询速度。

# 2.为经常需要排序、分组和联合操作的字段建立索引

经常需要ORDER BY、GROUP BY、DISTINCT和UNION等操作的字段，排序操作会浪费很多时间。如果为其建立索引，可以有效地避免排序操作。

# 3.为常作为查询条件的字段建立索引

如果某个字段经常用来做查询条件，那么该字段的查询速度会影响整个表的查询速度。因此，为这样的字段建立索引，可以提高整个表的查询速度。

# 4.限制索引的数目

==索引的数目不是越多越好。每个索引都需要占用磁盘空间，索引越多，需要的磁盘空间就越大==。修改表时，对索引的重构和更新很麻烦。越多的索引，会使更新表变得很浪费时间。

# 5.尽量使用数据量少的索引

如果索引的值很长，那么查询的速度会受到影响。例如，对一个CHAR(100)类型的字段进行全文检索需要的时间肯定要比对CHAR(10)类型的字段需要的时间要多。

# 6.尽量使用前缀来索引

如果索引字段的值很长，最好使用值的前缀来索引。例如，TEXT和BLOG类型的字段，进行全文检索会很浪费时间。如果只检索字段的前面的若干个字符，这样可以提高检索速度。

# 7.删除不再使用或者很少使用的索引

表中的数据被大量更新，或者数据的使用方式被改变后，原有的一些索引可能不再需要。数据库管理员应当定期找出这些索引，将它们删除，从而减少索引对更新操作的影响。

# 8.最左前缀匹配原则，非常重要的原则。

mysql会一直向右匹配直到遇到范围查询(>、<、between、like)就停止匹配，比如a1="1" and b="2" and c>=3 and d=4如果建立(a,b,c,d)顺序的索引，d是用不到索引的，如果建立(a,b,d,c)的索引则都可以用到，a,b,d的顺序可以任意调整。

# 9.=和in可以乱序。

比如a = 1 and b = 2 and c = 3 建立(a,b,c)索引可以任意顺序，mysql的查询优化器会帮你优化成索引可以识别的形式。

# 10.尽量选择区分度高的列作为索引。

区分度的公式是count(distinct col)/count(*)，表示字段不重复的比例，比例越大我们扫描的记录数越少，唯一键的区分度是1，而一些状态、性别字段可能在大数据面前区分度就是0，那可能有人会问，这个比例有什么经验值吗？使用场景不同，这个值也很难确定，一般需要join的字段我们都要求是0.1以上，即平均1条扫描10条记录。

# 11.索引列不能参与计算，保持列“干净”。

比如 from_unixtime(create_time) = ’2014-05-29’就不能使用到索引，原因很简单，b+树中存的都是数据表中的字段值，但进行检索时，需要把所有元素都应用函数才能比较，显然成本太大。所以语句应该写成create_time = unix_timestamp(’2014-05-29’);

# 12.尽量的扩展索引，不要新建索引。

比如表中已经有a的索引，现在要加(a,b)的索引，那么只需要修改原来的索引即可。

注意：选择索引的最终目的是为了使查询的速度变快。上面给出的原则是最基本的准则，但不能拘泥于上面的准则。读者要在以后的学习和工作中进行不断的实践。根据应用的实际情况进行分析和判断，选择最合适的索引方式。

# 二、MySQL如何创建索引

# 2.1使用 CREATE TABLE 语句

在使用 CREATE TABLE 语句创建表时，可以通过在字段后面添加索引类型来同时创建索引。具体语法如下：

创建单列索引：

CREATE TABLE table_name (
  column_1 data_type,
  column_2 data_type,
  ...
  column_n data_type,
  INDEX index_name (column_name)
);

例如，在创建 user 表时同时创建 name 字段的索引，可以使用以下语句：

  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50),
  age INT,
  INDEX idx_name (name)

# 2.2使用 CREATE INDEX 语句

MySQL 中创建索引的通用语法如下：

CREATE INDEX index_name ON table_name(column_name);
1

其中，index_name 为索引的名称，table_name 为要创建索引的表名称，column_name 为要创建索引的字段名。

例如，在 user 表中创建 name 字段的索引，可以使用以下语句：

CREATE INDEX idx_name ON user(name);
1

除了普通索引外，MySQL 可以创建唯一索引、主键索引、全文索引等不同类型的索引。这些索引的创建方式如下：

创建唯一索引：

CREATE UNIQUE INDEX index_name ON table_name(column_name);
1

其中，index_name 为索引的名称，table_name 为要创建索引的表名称，column_name 为要创建索引的字段名。

例如，在 user 表中创建 email 字段的唯一索引，可以使用以下语句：

CREATE UNIQUE INDEX idx_email ON user(email);
1

创建主键索引：

ALTER TABLE table_name ADD PRIMARY KEY (column_name);
1

其中，table_name 为要添加主键索引的表名称，column_name 为要添加主键索引的字段名。

例如，在 user 表中创建 id 字段的主键索引，可以使用以下语句：

ALTER TABLE user ADD PRIMARY KEY (id);
1

创建全文索引：

CREATE FULLTEXT INDEX index_name ON table_name(column_name);
1

其中，index_name 为索引的名称，table_name 为要创建索引的表名称，column_name 为要创建索引的字段名。

例如，在 article 表中创建 content 字段的全文索引，可以使用以下语句：

CREATE FULLTEXT INDEX idx_content ON article(content);
1

需要注意的是，在建立索引时需要根据具体情况来确定索引的类型及其列顺序，以达到最优化的查询效果。同时也需要注意索引过多会在一定程度上增加磁盘空间的占用，对数据的插入、删除和更新等操作性能产生一定影响，因此需要根据实际情况进行权衡。

# 2.3使用 ALTER TABLE 语句

在已经创建的表上添加索引，可以使用 ALTER TABLE 语句来完成。具体语法如下：

添加单列索引：

ALTER TABLE table_name ADD INDEX index_name (column_name);
1

其中，table_name 为要添加索引的表名称，index_name 为索引的名称，column_name 为要添加索引的字段名。

例如，在已经创建的 user 表中添加 name 字段的索引，可以使用以下语句：

ALTER TABLE user ADD INDEX idx_name (name);
1

添加联合索引：

ALTER TABLE table_name ADD INDEX index_name (column_1, column_2, ...);
1

其中，table_name 为要添加索引的表名称，index_name 为索引的名称，column_1、column_2 等为要添加索引的多个字段名，用逗号分隔。

例如，在已经创建的 user 表中添加 province 和 city 字段的联合索引，可以使用以下语句：

ALTER TABLE user ADD INDEX idx_province_city (province, city);
1

除了普通索引外，还可以添加唯一索引、主键索引、全文索引等不同类型的索引。这些索引的添加方式如下：

添加唯一索引：

ALTER TABLE table_name ADD UNIQUE INDEX index_name (column_name);
1

其中，table_name 为要添加索引的表名称，index_name 为索引的名称，column_name 为要添加索引的字段名。

例如，在已经创建的 user 表中添加 email 字段的唯一索引，可以使用以下语句：

ALTER TABLE user ADD UNIQUE INDEX idx_email (email);
1

添加主键索引：

ALTER TABLE table_name ADD PRIMARY KEY (column_name);
1

其中，table_name 为要添加主键索引的表名称，column_name 为要添加主键索引的字段名。

例如，在已经创建的 user 表中添加 id 字段的主键索引，可以使用以下语句：

ALTER TABLE user ADD PRIMARY KEY (id);
1

添加全文索引：

ALTER TABLE table_name ADD FULLTEXT INDEX index_name (column_name);
1

其中，table_name 为要添加索引的表名称，index_name 为索引的名称，column_name 为要添加全文索引的字段名。

例如，在已经创建的 article 表中添加 content 字段的全文索引，可以使用以下语句：

ALTER TABLE article ADD FULLTEXT INDEX idx_content (content);

# 三、覆盖索引

覆盖索引是一种特殊的索引查询技术，它能够通过索引==只取出所需的列，而无需再次访问数据表==，从而提高查询性能。

通常情况下，在使用索引进行查询时，即使使用覆盖索引也需要在数据表中找到匹配的行来获取其他需要的列，这会增加额外的 I/O 操作，降低查询性能。而使用覆盖索引则可以避免这种情况，因为所有查询的列都包含在索引中，因此可以直接从索引中获取所需的列，而无需读取数据表。

覆盖索引通常应用于以下情况：

  1. 查询的列都包含在索引中，不需要查询其他列。
  2. 数据表的数据较大，使用覆盖索引可以减少读取数据表的次数，从而提高查询性能。

例如，在 user 表中有 id、name、age、gender 四个字段，其中 id 字段已经创建了主键索引，如果要查询 name 和 age 两个字段的值，可以使用以下语句：

SELECT name, age FROM user WHERE id = 100;
1

如果没有覆盖索引，则需要根据 id 查找出匹配行，然后再从该行中获取 name 和 age 的值，这会产生额外的 I/O 操作。而如果在 name 和 age 字段上创建联合索引，则可以使用覆盖索引来优化查询：

CREATE INDEX idx_name_age ON user (name, age);

SELECT name, age FROM user WHERE id = 100;
1
2
3

这样就可以直接从索引中获取 name 和 age 的值，而无需再次访问数据表，从而提高了查询性能。

回表查询指的是在使用普通索引进行查询时，由于普通索引只包含了索引列和索引值，所以需要再次根据索引值去数据表中查询其他列的值，这样就需要多次访问数据表才能获取所有要查询的列的信息。

避免回表查询的原因主要有以下几点：

  1. 降低查询性能：回表查询需要多次访问数据表，会增加额外的 I/O 操作，从而降低了查询性能。
  2. 增加数据库负载：由于回表查询需要多次访问数据表，会增加数据库的负载，从而降低了数据库的处理能力。
  3. 浪费存储空间：需要存储更多的数据页和索引页，会占用更多的存储空间。

为了避免回表查询，可以使用覆盖索引、联合索引等技术来优化查询，或者使用覆盖索引和聚簇索引等组合索引来减少回表查询的次数。除此之外，还可以考虑对查询语句进行优化，例如通过拆分大查询为多个小查询等方式来避免回表查询。

# 四、什么情况下索引会失效 ?

  1. 在索引的字段上做任何的计算、函数等操作；

    EXPLAIN SELECT * FROM user WHERE age = age+1
    1

  2. 使用不等于（!=、<>）使索引失效

    EXPLAIN SELECT * FROM user WHERE age != 18
    1

  3. 使用like以通配符开头的模糊查询，其中**“%”在查询数据的左侧**时，使索引失效例如：“%小明”

    EXPLAIN SELECT * FROM user WHERE name like "%小"
    1

  4. 在查询语句中使用or连接之后，会使索引失效。哪怕是左右两个都是有索引的字段也会使索引失效。

    EXPLAIN SELECT * FROM user WHERE age=18  or name="小明"
    1

  5. 违反最左前缀法则，在查询语句中如果索引有多列，要遵守查询从索引的最左前列开始并且不跳过索引中的列。其中左侧的列可以使用索引，未加索引列之后的所有列索引失效。

    EXPLAIN SELECT * FROM user WHERE age=18  AND userId=18 and name="小明"
    1

  6. 字符串不加单引号，查询时使用一个没有加引号的字符串作为查询条件时，MySQL 会将该字符串按照数字类型来处理。

    EXPLAIN SELECT * FROM user WHERE age = '18';
    1

关于使用 or运算符

在 SQL 中，当查询语句中使用了 OR运算符，可能会导致索引失效。这是因为在使用 OR运算符时，数据库需要对多个条件进行判断，而这些条件可能会涉及到不同的索引。如果数据库优化器认为通过使用索引并不能提高查询效率，那么它就会选择全表扫描来执行查询操作，这时候就会导致索引失效。

举个例子，假设我们有一个包含用户 ID和用户年龄的用户表，并为这两个字段分别创建了索引。如果我们执行以下查询语句：

SELECT * FROM users WHERE user_id = 1 OR user_age > 30;
1

在上述查询语句中，我们使用了 OR运算符同时涉及到了 user_id和 user_age两个字段，而它们分别对应两个不同的索引。此时，如果数据库优化器认为通过两个索引来执行查询并不能提高效率，那么它就会选择全表扫描来执行查询操作，从而导致索引失效。

避免 OR运算符导致的索引失效可以通过以下几种方式来解决：

  1. 对查询语句进行重写，将 OR运算符转换为 UNION或者 UNION ALL 来执行。这样可以分别对涉及到的每个条件单独执行查询操作，并保证每个查询操作都能够使用相应的索引。
  2. 拆分查询语句，将涉及到的每个条件分开执行。例如，对于上述查询语句，我们可以拆分成两个查询语句，先执行 "SELECT * FROM users WHERE user_id = 1"，再执行 "SELECT * FROM users WHERE user_age > 30"，这样每个查询语句只涉及到一个条件，可以保证使用相应的索引进行查询。
  3. 通过对表结构进行调整，将涉及到的多个条件合并到一个索引中。例如，在上述用户表中，如果我们将 user_id和 user_age合并到一个联合索引中，那么在使用 OR运算符时，就可以直接使用该联合索引来执行查询操作。

总之，在 SQL 中，尽量避免使用 OR运算符可以有效减少索引失效的风险。如果必须使用 OR运算符，应该根据具体情况考虑以上几种方式来优化查询性能。

union和union all的区别 union：对两个结果集进行并集操作，不包括重复行，同时进行默认规则的排序；

union all：对两个结果集进行并集操作，包括重复行，不进行排序；
